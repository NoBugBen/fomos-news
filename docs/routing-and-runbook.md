# Routing and Runbook

## Chosen path

This repo now supports one evidence-based routing model in two modes:

- Local development: run the Rust API on `:3000` and the extracted Vite frontend on `:5173`, with Vite proxying `/api` to the Rust origin.
- Same-origin deployment: build the extracted frontend to `extracted/repo/fomos-news/dist/public` and let the Rust server serve that directory for non-`/api` routes.

This matches the current repository evidence:

- The frontend already calls relative `/api/...` paths.
- The extracted frontend is a Vite SPA with a static Express server, not an SSR app.
- The Rust backend already owns the real API surface and can safely remain the single deployed origin.

No extra reverse proxy, container stack, or OAuth-provider-specific infrastructure was added.

## Local development

1. Backend env:

```bash
cp .env.example .env
```

Keep these defaults unless you need to change ports:

```dotenv
APP_PORT=3000
FRONTEND_BASE_URL=http://localhost:5173
FRONTEND_DIST_DIR=extracted/repo/fomos-news/dist/public
```

2. Frontend env:

```bash
cd extracted/repo/fomos-news
cp .env.example .env
```

Default dev routing:

```dotenv
VITE_BACKEND_ORIGIN=http://127.0.0.1:3000
```

3. Run the backend from the repo root:

```bash
cargo run
```

4. Run the frontend from `extracted/repo/fomos-news`:

```bash
pnpm install
pnpm dev
```

5. Open `http://localhost:5173`.

Behavior:

- Browser requests for `/api/...` go to the Vite dev server first.
- Vite proxies those requests to `http://127.0.0.1:3000`.
- The browser still sees a same-origin `/api` path, which keeps the extracted frontend API calls unchanged.
- OAuth callback URLs generated by the frontend remain `http://localhost:5173/api/oauth/callback`, which the Vite proxy forwards to the Rust backend.

## Same-origin deployment

1. Build the frontend:

```bash
cd extracted/repo/fomos-news
pnpm install
pnpm build
```

2. Set backend env for the deployed public origin:

```dotenv
APP_ENV=production
APP_HOST=0.0.0.0
APP_PORT=3000
FRONTEND_BASE_URL=https://your-public-origin.example
FRONTEND_DIST_DIR=extracted/repo/fomos-news/dist/public
```

3. Run the Rust server from the repo root:

```bash
cargo run --release
```

Behavior:

- `/api/...` stays on the Rust backend.
- `/` and other SPA routes serve the built frontend from `FRONTEND_DIST_DIR`.
- Deep links such as `/briefing` fall back to `index.html`.

## Supported paths

- Supported now:
  - `GET /api/healthz`
  - `GET /api/news`
  - `GET /api/news/:id`
  - `GET /api/briefings/latest`
  - `GET /api/briefings`
  - `GET /api/briefings/:date`
  - `POST /api/subscribe`
  - `GET /api/auth/session`
  - `POST /api/auth/logout`
  - `GET /api/oauth/callback`
  - SPA routes served by Rust only when `FRONTEND_DIST_DIR` exists
- Not added here:
  - External reverse-proxy configs such as Nginx/Caddy
  - Docker or orchestration files
  - Real OAuth provider credentials or provider-specific callback setup

## Notes

- `SESSION_COOKIE_NAME` still defaults to `fomos_news_session`. Before wiring the extracted frontend auth/session UI more deeply, align it to `app_session_id` if you want parity with the extracted shared constant.
- The extracted frontend also has optional map and OAuth-related env vars. Leave them blank unless you have real values.
- The old extracted Express static server still exists upstream, but this repo's supported combined deployment path is now the Rust server as the single origin.
